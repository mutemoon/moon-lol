# MoonLOL Desktop — 产品设计文档

## 一、产品定位

MoonLOL Desktop 是一个基于 Bevy 引擎（Rust ECS）的《英雄联盟》复刻项目的桌面客户端工具，核心目标是提供一个 **AI Agent 对战模拟与调试平台**。用户可以在 Summoner's Rift 上配置不同类型的 AI Agent 进行自动化对战，实时观察 Agent 的决策过程，并回溯历史对局数据，用于 AI 策略研究与迭代。

---

## 二、Agent 体系

### 2.1 Agent 类型

支持三种模式的 Agent，覆盖从规则驱动到智能决策的完整光谱：

| 类型 | 驱动方式 | 适用场景 |
|---|---|---|
| **LLM Agent** | 大语言模型（如 Gemini、DeepSeek） | 自然语言策略推理、复杂决策、探索性实验 |
| **RL Agent** | 强化学习模型（PPO 等） | 高频微操、补刀优化、机械性任务 |
| **Script Agent** | 用户编写的 JavaScript 脚本 | 确定性行为树、固定套路、基准测试对手 |

### 2.2 各类型 Agent 的运行时能力

#### LLM Agent
- 通过 WebSocket 接收游戏观测数据（`get_observe`），调用 LLM 推理后下发动作指令（`action`）
- Desktop 端实时展示 Agent 的**思考链（Chain of Thought）**，以对话气泡的形式呈现：
  - 🟢 **系统观测**（User 消息）：当前局势数据摘要
  - 🔵 **模型推理**（Assistant 消息）：思考过程、决策依据、最终动作
- 支持配置系统级 Preamble Prompt（全局策略指导）和英雄级 Prompt（角色个性化微调）
- 支持选择模型、调节思考深度（Thinking Depth）

#### RL Agent
- 通过 Gymnasium 环境接口（`MoonLoLEnv`）与游戏引擎交互
- Desktop 端展示训练/推理过程中的关键指标：
  - **Reward 分项**：补刀奖励、击杀奖励、死亡惩罚、位置奖励等各分项 reward
  - **策略分布**：当前动作空间的概率分布可视化
  - **Value 估计**：Critic 网络对当前状态的价值评估
  - **训练曲线**：Episodic Return、Loss、KL 散度等实时图表
- 支持加载已训练的模型权重（`.pth`），也支持在线训练

#### Script Agent
- 使用 **JavaScript** 作为脚本语言，通过内嵌 JS 运行时（如 Boa / QuickJS）在 Rust 侧执行
- 内置脚本编辑器（Monaco Editor），原生支持 JS 语法高亮、智能补全和类型提示
- 支持脚本的**实时热重载**：修改脚本后无需重启对局
- 提供调试工具：断点、单步执行、变量观察
- 暴露游戏 API 绑定（`observe()`、`action()`、`log()` 等），脚本可直接调用
- 内置常用脚本模板：补刀脚本、走A脚本、技能连招脚本

### 2.3 预设 Agent 系统

- 每种 Agent 类型都支持创建**预设 Agent（Preset）**
- **Agent 预设是"策略大脑"，英雄无关**：只含 Agent 类型、Prompt、模型/前言等"怎么思考"的配置，**不绑定具体英雄**。同一套"激进压制"大脑可被锐雯、菲奥娜等任意英雄复用。
- 英雄绑定交给**英雄预设**（§3.1）负责——英雄预设引用一个 Agent 预设作为大脑。
- 预设内容包含：Agent 类型、Prompt 配置、关联的模型名（LLM）/脚本文件（Script）/权重（RL）、英雄级前言（preamble）
- 预设以 JSON 文件持久化存储（`~/.moon-lol/agent_presets.json`），支持导入/导出分享

## 三、对局配置系统

### 3.0 设计原则：配置页只做编排，不做内联编辑

对局配置页的职责是**从已定义好的预设中"编排"出一局对战**，而不是逐项配置细节。这是与早期 MVP（把英雄/Prompt/出生点全部内联在一张表单里）的关键区别。

- **编排页保持轻量聚焦**：只展示"谁打谁、从哪出生、怎么算赢"这一层信息，不展开 Agent 的 Prompt、模型权重、超参数等深度字段，避免页面信息过载、失去重点。
- **复用走"预设 + 下拉选择"**：Agent 配置、出生点坐标这类可复用资产统一封装为**预设（Preset）**，在编排页通过下拉框选择。
- **下拉框内置"新建"入口**：每个预设下拉框的末尾都有一个 **＋ 新建预设** / **管理预设** 入口，点击后跳转到对应的**独立管理页面**完成深度编辑，再回到编排页继续。这是创建/编辑预设的唯一入口——编排页本身不提供内联的 Prompt 编辑器、坐标输入框等。
- **三类资产分层管理**：
  | 资产 | 在编排页的角色 | 深度配置位置 |
  |---|---|---|
  | **英雄预设**（§3.1） | 每个阵容槽位的**唯一选择单元** | 英雄预设管理页 |
  | **Agent 预设**（§2.3） | 被英雄预设引用为"大脑" | Agent 预设管理页 |
  | **出生点预设**（§3.2） | 被英雄预设引用为坐标 | 出生点预设管理页 |
  | **胜利条件**（§3.3） | 选择预设模板或自定义 | 胜利条件构建器 |

> **层级关系**：编排页槽位 → 英雄预设 → { 英雄 + Agent 预设（脑）+ 出生点预设（坐标）}。Agent 预设与出生点预设不直接在编排页选择，而是由英雄预设组合引用。预设的增删改逻辑都在各自的**独立管理页面**完成，**不**在编排页内做重复的内联 CRUD。
>
> **例外：编排页槽位允许"临时覆盖"**——拖入英雄预设后，仍可在槽位上**临时改写**其绑定的 Agent 预设 / 出生点预设（用于这一局的对战调试，不希望污染原预设）。一旦改动任一项，槽位即与原预设**解绑**（仅保留展开后的具体值），并提供「存为新预设」按钮把这份临时配置固化成新的英雄预设。原预设永不被原地修改。详见 §3.1.1。

### 3.1 英雄预设与阵容配置

**英雄预设**是编排页槽位的唯一选择单元，把"打什么英雄、用什么大脑、从哪出生"三者打包成一个可复用单元：

- 字段：`name`（命名）、`champion`（英雄）、`agent_preset_name`（引用的 Agent 预设，即大脑）、`spawn_preset_name`（引用的出生点预设）
- 在英雄预设管理页绑定这三者；其中 Agent 预设、出生点预设通过下拉选择，下拉末尾有「＋ 新建」跳转对应管理页
- 持久化到 `~/.moon-lol/hero_presets.json`，支持导入/导出

**阵容配置**（编排页）：

- **双阵营配置**：Order（秩序/蓝方）与 Chaos（混沌/红方）
- 每个阵营支持添加**任意数量**的槽位
- 每个槽位只选**一个英雄预设**——下拉选择，末尾「＋ 新建英雄预设」跳转英雄预设管理页
- 槽位下方展示展开预览：英雄名 · Agent 类型（如 `锐雯 · LLM`）
- **不在编排页内联编辑**：英雄、Agent 类型、Prompt、模型权重、出生点坐标等深度字段，全部分散在英雄预设/Agent 预设/出生点预设内部，各自在独立管理页维护
- 保存场景时，前端把所选英雄预设展开为具体的 `{champion, prompt, spawn_point, agent_type}` 写入场景；后端场景格式与 RON 生成逻辑不变

#### 3.1.1 拖入英雄预设后的临时操作

配置一局对战时，围绕"拖进来的英雄预设"有三类高频诉求，编排页槽位原生支持，**不**迫使玩家反复去管理页改预设：

| 诉求 | 交互 | 行为 |
|---|---|---|
| **重用上次配置** | 把英雄预设拖/选进槽位 | 槽位即代表一份完整配置（英雄 + 大脑 + 出生点） |
| **临时改 Agent 预设或出生点** | 槽位下方 Agent / 出生点选择器始终展开，初始显示继承值并打「继承」标记 | 改动任一项 → 槽位与原预设**解绑**，标记变「已覆盖」，但具体值保留可用，原预设不被修改 |
| **编辑预设本身** | 槽位「✏ 编辑」按钮 | 跳 `/heroes?edit=<name>`，英雄预设管理页以该条目进入编辑态（仅绑定态可见） |
| **临时配置另存为新预设** | 解绑后槽位出现「存为新预设」按钮 | 弹窗预填自动命名（`<原预设名> · 副本`），确认后槽位重新绑定到新预设、恢复「继承」态 |

- **设计意图**：编排页是"做实验"的地方，玩家常想"这局让锐雯换个大脑试试，但不想动我那套激进压制预设"。临时覆盖 + 另存为新预设覆盖了这一闭环，而**原地编辑原预设**永远走管理页，避免误改破坏其他引用该预设的场景。
- **解绑语义**：解绑后 `heroPresetName` 置空、`dirty=true`；槽位只持有展开后的具体值（champion / agentPresetName / spawnPresetName）。保存场景时按具体值写入，与普通槽位无差异。
- 另存为新预设时若命名与既有预设重名，自动追加 `· 副本 2 / 3 / ...` 后缀，避免覆盖。

### 3.2 出生点预设

- 出生点统一封装为**预设**，在编排页通过下拉框选择，不在编排页内联输入坐标或弹地图。
- 内置常用位置预设：
  - 上路一塔前方、中路河道、下路草丛、蓝方 Buff 区等
- **出生点预设管理页**负责预设的全部 CRUD：
  - 通过 SVG 地图点选器交互式选取坐标（15000×15000 游戏坐标系）
  - 命名保存、编辑、删除
  - 支持按阵营（Order / Chaos）筛选
- 编排页的下拉框末尾提供「＋ 新建预设」入口，跳转到该管理页。

### 3.3 胜利条件配置

支持灵活的胜利条件系统，可自由组合形成复杂判定逻辑：

#### 基础条件原子

| 条件类型 | 参数 | 示例 |
|---|---|---|
| 补刀数达标 | `minion_kills >= N` | 谁先补 10 个小兵 |
| 击杀英雄 | `kills >= N` | 拿下一血 / 三杀获胜 |
| 摧毁防御塔 | `turret_destroyed(lane, tier)` | 推掉上路一塔 |
| 摧毁水晶 | `inhibitor_destroyed(lane)` | 摧毁任意水晶 |
| 摧毁基地 | `nexus_destroyed` | 推掉主基地 |
| 经济领先 | `gold_lead >= N` | 经济差达到 2000 |
| 存活时长 | `survive_duration >= T` | 存活超过 5 分钟 |
| 总伤害量 | `total_damage >= N` | 累计输出 10000 伤害 |

#### 逻辑组合

- **AND**：所有子条件同时满足（如"补 10 个兵 **且** 不死"）
- **OR**：任一子条件满足（如"拿一血 **或** 先补 10 个兵"）
- **NOT**：条件取反（如"**不允许** 使用技能"）
- 支持嵌套组合，形成任意深度的复杂胜利条件树

#### 预设胜利条件

- **速战速决**：先拿一血获胜
- **补刀竞赛**：先补 10 个小兵获胜
- **推塔竞速**：先推掉对方一塔获胜
- **综合实力**：推掉主基地获胜（标准模式）
- 用户可创建自定义胜利条件预设

### 3.4 对局场景（Scenario）

- 完整的对局配置（阵营 Agent + 出生点 + 胜利条件）可保存为**场景预设**
- 场景以命名的 JSON 文件持久化，支持从侧边栏快速加载
- 支持从历史对局一键复刻场景配置

## 四、对局运行与调试

### 4.1 实时调试面板

对局启动后进入调试视图，提供以下实时信息：

- **连接状态**：WebSocket 通道连接状态指示
- **控制台日志**：游戏引擎运行日志，支持按实体 ID 过滤
- **AI 决策思维**：LLM Agent 的完整对话历史流，实时滚动更新
- **RL 指标面板**：强化学习 Agent 的 reward、loss 等实时图表

### 4.2 调试控制

- **上帝模式**：无敌 + 全视野
- **无冷却时间**：所有技能冷却归零
- **暂停/恢复**：冻结游戏时间
- **重置位置**：将英雄传送回初始位置
- **切换英雄**：调试视角切换到不同英雄
- **命令行测试床**：手动发送 WebSocket 指令调试

### 4.3 回放与复盘

- 对局结束后，完整对话历史自动持久化存储
- 历史对局页面支持：
  - 按时间线浏览所有已完成的对局
  - 查看每个 Agent 的完整决策对话记录
  - 查看系统级和英雄级 Prompt 配置
  - 删除历史记录
