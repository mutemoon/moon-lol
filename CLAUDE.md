# 测试思考逻辑

## 总原则

- 优先写“系统行为测试”，不要优先写“常量抄写测试”。
- 如果一个功能在真实运行时依赖 Bevy 的 `App`、插件、时间推进、事件、资源或 ECS 查询，那么测试也应该尽量经过这条真实路径。
- 对 `skill`、`action`、`controller`、`movement` 这类系统，测试目标不是验证“代码里写了什么”，而是验证“玩家操作后世界状态会变成什么”。

## 技能测试原则

- `skill` 测试应尽量像 `minion` 测试一样，运行在无头 Bevy 实例里。
- 测试必须显式初始化场景：至少包含施法者、敌方目标、友方目标、必要的 `Transform`、`Team`、`Health`、`Skills`、`SkillPoints`、`AbilityResource` 等组件。
- 测试必须显式注册技能依赖的资产：如 `SpellObject`、`SkillEffect`，不要依赖隐式全局状态。
- 测试应通过事件或输入链触发技能，而不是直接手工调用内部函数后再断言局部变量。
- 优先走真实输入语义：例如 `CommandAction -> Action::Skill`、`Action::SkillLevelUp`，而不是绕开行为层直接修改组件。

## 为什么不能只做静态断言

- 只验证 `DashMoveType::Fixed(250.0)`、`DamageShape::Circle { radius: 300.0 }` 这类断言，最多只能证明测试作者重新写了一遍常量。
- 这类测试无法发现：
- 冷却没有启动。
- 蓝耗没有扣除。
- 冷却中仍然可以重复施法。
- 范围伤害打到了友军。
- 场景里目标筛选逻辑失效。
- 插件没接上，导致事件根本没被消费。

## 一个合格的技能测试应覆盖什么

- 初始场景：谁是施法者，谁是敌人，谁是友军，谁在范围内，谁在范围外。
- 输入：玩家进行了什么操作，例如按技能键、加点、指定目标点。
- 帧推进：经过多少帧、多少秒后，系统应该产生什么变化。
- 结果：蓝量、冷却、技能等级、位置、血量、Buff、MovementState、事件副作用是否符合预期。

## 时间推进原则

- 对依赖 `FixedUpdate` 或 `Time` 的系统，不要只 `app.update()` 一次就断言最终状态。
- 测试里应使用固定帧率和手动时间推进，例如 `TimeUpdateStrategy::ManualDuration`。
- “冷却结束”“位移完成”“持续伤害结算”这类逻辑，必须通过多帧推进验证。

## 场景构建原则

- 测试场景要尽可能小，但必须完整。
- 能只启必要插件时，就不要加载整套运行时插件。
- 但如果功能正确性依赖插件间联动，就必须把依赖链一起挂上，不能偷省。
- 例如技能测试至少应考虑：
- `PluginAction`
- `PluginSkill`
- `PluginCooldown`
- `PluginDamage`
- `PluginLife`
- 其他是否启用，取决于该技能是否依赖 `Movement`、`Buff`、`Controller`、`AttackReset` 等链路。

## 输入模拟原则

- 测试“用户行为”时，要模拟用户语义，不要直接改内部状态。
- 可接受的方式：
- 发送 `CommandAction`。
- 发送 `Action::Skill` / `Action::SkillLevelUp`。
- 在确实需要时，构造 `ButtonInput<KeyCode>`、窗口位置、光标点位，测试更上层控制器逻辑。
- 不推荐的方式：
- 直接修改 `CoolDown.timer` 后说“冷却逻辑正确”。
- 直接扣 `AbilityResource.value` 后说“蓝耗正确”。

## 断言原则

- 断言优先观察“世界状态”，不要优先观察实现细节。
- 好断言：
- 技能释放后蓝量减少。
- 技能释放后冷却开始。
- 冷却未结束前再次施法失败。
- 敌方目标掉血，友方目标不掉血。
- 差断言：
- 某个 enum 变体存在。
- 某个常量等于某个数字，但这个数字本身就是测试里手写的。

## 什么时候允许保留纯数学测试

- 纯函数、纯几何、纯数值映射可以保留小而快的单元测试。
- 例如路径投影、半径插值、伤害公式、护甲减伤曲线。
- 但这类测试只能作为补充，不能代替系统集成测试。

## 推荐结构

- 为复杂系统建立专门的 harness。
- harness 应负责：
- 创建无头 `App`
- 安装必要插件
- 注册测试资产
- 生成初始实体
- 提供“模拟输入”和“推进时间”的辅助方法
- 提供“查询世界状态”的辅助方法
- 这样每个测试只描述：
- 给定什么场景
- 玩家做了什么
- 最终应该发生什么

## 已落地经验

- `tests/skill_integration.rs` 是当前推荐方向：
- 使用无头 Bevy `App`
- 注册 `SpellObject` 和 `SkillEffect`
- 通过 `CommandAction` 模拟玩家输入
- 通过手动时间推进验证冷却
- 通过场景目标验证技能伤害筛选

- `tests/riven_integration.rs` 进一步验证了另一条更重要的经验：
- 如果本地已经有从当前 LoL 版本导出的真实资源，英雄技能正确性测试应优先使用这些真实资源。
- 但不要直接猜某个技能路径会对应哪个 `assets/data/*.lol` 文件。
- 导出后的 `assets/data/*.lol` 是按 wad hash 落盘的，技能配置通常不一定独立存在为“每技能一个文件”。
- 更稳的做法是：
- 从英雄主 prop 包加载，例如 `DATA/Characters/Riven/Riven.bin`
- 从 prop 包内遍历 `SpellObject`
- 再按 `m_script_name` / `object_name` 选出目标技能
- 这样测试资源来源是真实的，定位方式也与导出结构一致

## 真实资源测试策略

- 如果目标是验证“英雄技能是否正确”，优先使用本地已导出的真实资源。
- 如果目标是验证“系统行为是否正确”，可以继续使用最小手工资源。
- 两者不要混为一类测试。

### 优先级

- 英雄技能行为正确性测试：
- 优先使用真实导出资源
- 例如从 `DATA/Characters/Riven/Riven.bin`、`DATA/Characters/Fiora/Fiora.bin` 里提取真实 `SpellObject`

- 通用系统行为测试：
- 使用最小手工资源
- 例如蓝耗门禁、冷却门禁、范围筛选、Buff 链路

- 最新版本兼容性检查：
- 允许依赖当前本地导出的全量资源
- 但这类测试要接受上游资源变动带来的不稳定性

## 从真实资源取技能的规则

- 不要假设 `Characters/X/Spells/...` 一定能直接映射到某个 `assets/data/{hash}.lol` 文件。
- 优先从英雄主 bin 入手，例如：
- `DATA/Characters/Riven/Riven.bin`
- `DATA/Characters/Fiora/Fiora.bin`
- `DATA/Characters/Hwei/Hwei.bin`

- 从主 bin 中：
- 解析 `PropFile`
- 过滤 `SpellObject` 类型条目
- 再按 `m_script_name`、`object_name` 精确筛选技能

- 这样做的原因：
- 导出阶段保留的是哈希文件名，不保留原路径文件树
- 技能资源可能被打包进英雄主配置，而不是独立文件
- 主 bin 的结构更稳定，也更贴近英雄真实运行时配置

## 资源路径和 hash 约束

- 运行时对 prop 路径做 hash 时，默认使用小写路径。
- 如果测试里需要按路径去定位导出文件，必须和运行时保持同样的小写规则。
- 但即使如此，也不要优先依赖“路径直接命中文件”这件事；优先走主 bin 反查。

## 断言真实资源时的注意事项

- 不要硬编码“某技能一定耗蓝”“某技能一定有某个字段”。
- 真实资源里字段可能为空，例如 `mana: None`。
- 测试应优先读取真实资源，再根据真实值断言行为。
- 也就是说：
- 可以断言“释放后 mana 按资源中的 cost 扣除”
- 不要断言“释放后一定扣 20 蓝”

## 对技能测试的推荐分层

- `*_integration.rs`
- 使用无头 Bevy + 真实英雄资源
- 验证该英雄技能的真实行为

- `skill_integration.rs`
- 使用无头 Bevy + 最小手工资源
- 验证技能系统公共行为

- 如果以后需要兼容性检查：
- 再增加一层面向最新导出资源的检查测试
- 但不要让所有基础开发都绑定在这层之上

## 写测试时的自检问题

- 这个测试是在验证真实行为，还是在重复抄实现细节？
- 如果技能插件失效，这个测试会失败吗？
- 如果事件链断了，这个测试会失败吗？
- 如果目标筛选错了，这个测试会失败吗？
- 如果冷却和蓝耗没生效，这个测试会失败吗？
- 如果这些问题的答案大多是否，那么测试还不够好。
