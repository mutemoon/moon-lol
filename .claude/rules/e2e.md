**E2E 测试代码定位器规范**

**原因 (Why)**

- **防范样式变更**：首选 `data-testid` 定位元素，避免因 UI 样式或 DOM 结构调整导致用例失效。
- **避免易变属性**：不使用自增 ID、随机 Hash 或数据库主键这类在不同测试环境中不稳定的标识。

**示例 (Examples)**

- **普通元素**：
  - 模板：`<button data-testid="submit-btn">`
  - 测试：`page.getByTestId('submit-btn')`
- **列表元素 (方案一：容器 + 统一 TestID + 过滤)**：
  - 模板：容器设 `data-testid="room-list"`，每项均设 `data-testid="room-item"`
  - 测试：`page.getByTestId('room-list').getByTestId('room-item').filter({ hasText: '目标名称' })` 或 `.nth(0)`
- **列表元素 (方案二：稳定业务标识)**：
  - 模板：`<div :data-testid="`room-item-${room.name}`">` (适用于 `room.name` 绝对稳定唯一的情况)

**E2E 测试代码质量规范**

**状态隔离与清理**

- 测试必须假设后端数据会持久化；不要依赖“空数据库”“首次运行”或固定账号天然干净。
- 创建测试数据时使用本次运行唯一的业务名称；如果业务允许，优先使用唯一账号、唯一房间名、唯一 agent 名等隔离维度。
- 对会污染全局状态的操作必须清理，例如匹配队列入队后要出队、创建房间后要解散、创建 agent 后要删除。
- 清理逻辑应放在 `try/finally` 或 fixture teardown 中，避免中途断言失败后留下脏数据。
- 多 browser context / page 的测试必须在 `finally` 中关闭 context，避免资源泄漏影响后续用例。

**断言与定位精度**

- 断言要绑定当前测试创建的数据或当前业务容器，避免使用页面全局的宽泛文本匹配。
- 列表、下拉选项、卡片等重复元素应先 scope 到稳定容器，再用 `filter({ hasText })`、业务名或 testid 定位。
- 避免只断言通用文案，例如 `已保存`、`v1`、`top_solo`；这类文案可能来自旧数据或无关元素。
- URL 断言应尽量保留并复用当前流程产生的 id，避免只验证“形状正确”而没有验证进入的是目标资源。

**流程粒度与可维护性**

- 单个 E2E 用例应覆盖一个清晰用户路径；不要把登录、创建、发布、匹配、房间、观战、清理全部塞进一个超长用例，除非它明确是端到端冒烟测试。
- 重复流程应抽为 helper 或 fixture，例如登录、创建房间、创建并发布 agent、确认弹窗。
- 测试标题要描述真实行为和业务结果，而不是泛泛写 “without error”。
- 注释应解释业务意图、测试风险或历史 bug 背景；不要逐行复述代码动作。

**并行与重试安全**

- Playwright 本地可能并行执行，CI 也可能重试；测试必须能在并行、重试、重复运行时保持稳定。
- 固定手机号、固定业务名、固定 snapshot 版本等共享状态容易产生串扰；若无法避免，必须在测试前显式重置或在测试后可靠清理。
- 对异步后端状态变化使用 Playwright 的自动等待断言，不使用硬编码 sleep。
