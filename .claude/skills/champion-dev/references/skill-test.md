# 测试设计原则

- 优先系统测试：不要优先写常量抄写测试。如果逻辑依赖 Bevy 应用程序、插件、时间流转、事件或实体查询，测试也必须运行真实逻辑路径。
- 状态变化断言：测试的目标是验证玩家操作后世界状态的变化，而不是验证代码的具体编写方式。
- 公共校验测试：技能系统的公共校验逻辑在核心技能包内测试，包含冷却、资源限制和重施等通用网关的验证。

# 英雄技能测试架构

- 测试归属：各英雄技能的测试代码位于对应的英雄目录下，属于条件编译模块。
- 统一测试入口：集成测试和渲染测试统合在测试文件中，不再拆分渲染测试文件。
- 测试模式支持：测试默认运行在无头 Bevy 实例中，可通过测试辅助工具切换无头模式和渲染模式。

# 测试辅助工具运行机制

- 辅助工具命名：统一使用测试包中的 `ChampionTestHarness`，避免英雄模块重复编写测试环境代码。
- 英雄特有配置：每个英雄模块只需要提供包含配置路径、皮肤路径、插件注册回调、被测组件类型等的 `ChampionHarnessConfig`。
- 无头模式：使用模拟的技能句柄以保证加载的高效同步，不需要依赖显卡。通过启动系统直接在世界中生成施法者和测试目标。
- 渲染模式：使用真实的配置文件和皮肤描述文件，通过资产服务器加载网格模型和动画，在完成资产加载后推进测试。
- 录像生成：在渲染模式下，测试运行结束时会调用结束方法，将帧图输出并自动合成视频存放在对应的测试视频路径下。

# 技能测试覆盖要求

- 场景初始化：显式构造测试所需的施法者、盟友或敌方目标，赋予初始的位置、队伍属性、血量、技能列表、蓝量等组件。
- 玩家施法输入：使用派发指令模拟真实施法输入，不要通过手动修改组件值来模拟施法。
- 时间手动推进：如果系统逻辑依赖固定更新或时间组件，必须使用固定帧率和手动时间流转方法。
- 断言关注点：优先断言蓝量扣减、冷却状态变化、受击单位血量、状态添加以及最终位置，而不是断言枚举类型或硬编码的常量。

# 真实资源测试规范

- 资源检索：不建议硬编码提取的技能数据路径，建议从英雄的主描述配置文件入手，解析其属性定义，再过滤得到目标技能的配置对象。
- 属性校验：不要硬编码断言某些技能一定会扣除固定蓝量或带有特定属性，而是应该首先读取当前技能资产的真实配置，再根据资产配置的属性数值断言世界状态的变化。
