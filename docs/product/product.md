# MoonLOL — 产品设计文档

> 配套技术规范：[接口设计文档](./docs/API_DESIGN.md)

## 一、产品定位与三形态总纲

### 1.1 产品定位

MoonLOL 是一个基于 Bevy 引擎（Rust ECS）的《英雄联盟》复刻项目之上的 **AI Agent 电竞平台**。用户扮演"电竞经理"的角色：配置、培养、发布 Agent（英雄 + 策略大脑），让 Agent 在 Summoner's Rift 上自动对战、自动上分。平台同时服务三类人群：

- **研究者**：调试 Agent 策略、做对照实验、回溯决策链
- **普通玩家**：在公共房间社交观战、引入社区 Agent 组局
- **竞技作者**：打磨 Agent 冲天梯、追求排名

### 1.2 三种对局形态（长期并存）

平台支持三种对局形态，**长期并存、各有侧重**，且共享同一套 Agent 资产（预设库、配置、社区）：

| 形态 | 游戏运行位置 | 登录 | Agent 来源 | 侧重点 |
|---|---|---|---|---|
| **A. 本地多 Agent 对局** | **本机**（desktop 独占） | 可不登录 | 本地预设库 | 研究调试 |
| **B. 房间多用户对局** | 服务器 | 必须登录 | 各成员自带 Agent | 社交娱乐 |
| **C. Rank 竞技** | 服务器 | 必须登录 | 各自发布的参赛快照 | 竞技排名 |

### 1.3 形态演进与依赖关系

```
本地调试（desktop 独占，可离线）
        │
        │  共享 Agent 资产 / 服务器化
        ▼
房间多用户对局（社交，实时观战）
        │
        │  复用房间的对局编排 / 服务器托管能力
        ▼
Rank 竞技（全自动 7×24 排队上分）
```

- **房间是 rank 的基础**：rank 本质上是"系统当房主、全自动匹配的房间"，复用同一套服务器对局托管与观战管线。
- **Desktop 是 Web 的超集**：Web 端是轻量消费端；Desktop 保留为"专业调试版"，Web 能做的（房间/rank/观战）Desktop 都能做，额外拥有本地离线调试能力。

### 1.4 客户端形态

| 客户端 | 定位 | 能力范围 |
|---|---|---|
| **Desktop（Tauri + Vue）** | 专业调试版 + 全功能 | 本地对局（独占） + 房间 + Rank + 观战 + 完整调试面板 |
| **Web（浏览器）** | 轻量消费端 | 房间 + Rank + 观战 + Agent/预设管理（不含本地对局） |

> Desktop 本地对局以**本地预设库**为主、可离线运行；登录后可手动把 Agent/场景等配置同步到云端，跨设备复用。

## 二、Agent 体系

### 2.1 核心术语：Agent 配置 → Agent → 参赛快照

为避免概念混淆，文档统一以下三层术语（注意：旧文档的"英雄预设"现已更名为"Agent"，旧文档的"Agent 预设"更名为"Agent 配置"）：

```
┌─────────────────────────────────────────────────────────────┐
│  Agent 配置（Agent Config） = "策略大脑"，英雄无关            │
│    字段：类型（LLM/RL/Script）、Prompt、模型/权重、前言……    │
│    例：「激进压制」「稳健补刀」——可被任意英雄复用            │
└─────────────────────────────────────────────────────────────┘
                          │ 绑定
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Agent = 英雄 + Agent 配置 + 出生点 = 独立"选手"             │
│    是参加对局、拥有 ELO、出现在排行榜上的主体                │
│    例：「锐雯 · 激进压制」「菲奥娜 · 稳健补刀」              │
└─────────────────────────────────────────────────────────────┘
                          │ 发布（冻结）
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  参赛快照（Snapshot）= Agent 在某时刻的不可变冻结版本        │
│    用于：Rank 队列参赛 / 分享到社区被他人引入                │
└─────────────────────────────────────────────────────────────┘
```

**关键规则**：
- **ELO 主体是 Agent**，不是用户。用户是电竞经理，可养多个 Agent。
- ELO 按 **Agent × 模式** 分别计算（同一 Agent 的上单 ELO 与中单 ELO 互不影响）。
- **进行中的对局使用参赛快照**：用户修改 Agent 配置不影响进行中的局，下次发布/参赛才生效。
- **一个 Agent 可报名多个模式**（上单 SOLO、未来中单 SOLO……），各模式独立 ELO。

### 2.2 Agent 类型

支持三种 Agent 类型，覆盖从规则驱动到智能决策的完整光谱。Agent 配置按类型携带不同字段：

| 类型 | 驱动方式 | 适用场景 |
|---|---|---|
| **LLM Agent** | 大语言模型（Gemini / DeepSeek 等） | 自然语言策略推理、复杂决策、探索性实验 |
| **RL Agent** | 强化学习模型（PPO 等） | 高频微操、补刀优化、机械性任务 |
| **Script Agent** | 用户编写的 JavaScript 脚本 | 确定性行为树、固定套路、基准测试对手 |

### 2.3 各类型 Agent 的运行时能力

#### LLM Agent
- 通过 WebSocket 接收游戏观测数据（`get_observe`），调用 LLM 推理后下发动作指令（`action`）
- Desktop 端实时展示 Agent 的**思考链（Chain of Thought）**，以对话气泡的形式呈现：
  - 🟢 **系统观测**（User 消息）：当前局势数据摘要
  - 🔵 **模型推理**（Assistant 消息）：思考过程、决策依据、最终动作
- 支持配置系统级 Preamble Prompt（全局策略指导）和英雄级 Prompt（角色个性化微调）
- 支持选择模型、调节思考深度（Thinking Depth）

#### RL Agent
- 通过 Gymnasium 环境接口（`MoonLoLEnv`）与游戏引擎交互
- Desktop 端展示训练/推理过程中的关键指标：
  - **Reward 分项**：补刀奖励、击杀奖励、死亡惩罚、位置奖励等各分项 reward
  - **策略分布**：当前动作空间的概率分布可视化
  - **Value 估计**：Critic 网络对当前状态的价值评估
  - **训练曲线**：Episodic Return、Loss、KL 散度等实时图表
- 支持加载已训练的模型权重（`.pth`）；**在线训练由用户自行进行**，平台只提供接入接口作为 Agent 动力

#### Script Agent
- 使用 **JavaScript** 作为脚本语言，通过内嵌 JS 运行时（如 Boa / QuickJS）在 Rust 侧执行
- 内置脚本编辑器（Monaco Editor），原生支持 JS 语法高亮、智能补全和类型提示
- 支持脚本的**实时热重载**：修改脚本后无需重启对局
- 提供调试工具：断点、单步执行、变量观察
- 暴露游戏 API 绑定（`observe()`、`action()`、`log()` 等），脚本可直接调用
- 内置常用脚本模板：补刀脚本、走A脚本、技能连招脚本

### 2.4 Agent 的所有权、可见性与社区

- **全局预设库**：所有 Agent / Agent 配置 / 出生点 / 场景统一管理，跨形态复用。本地对局用本地副本，房间/Rank 用云端副本。
- **可见性**（用户设置）：`私有` / `好友可见` / `公开`。
- **Fork 与上游同步**：公开的 Agent 可被他人 Fork 为自己的副本；Fork 后可"从上游拉取最新配置"同步更新。
- **社区快照引入**（Q9）：用户可把某个 Agent 的参赛快照分享到社区，其他用户可引入作为自己的 Agent 参赛。
- **可见性即权限**：导出/查看 Prompt / 模型配置等敏感字段的权限，由该 Agent 的可见性设置决定（详见 §6.3）。

### 2.5 配置快照与版本（Rank 专用）

Rank 队列里用户可能不在线，Agent 配置变更必须显式生效，避免"对局进行中配置被改"的不一致：

- **发布（Publish）**：用户手动把 Agent 的当前配置**发布为一个参赛快照**，记为 `v1 / v2 / v3 …`。
- **队列取用规则**：Rank 匹配队列**始终使用该 Agent 最新发布的那一份快照**。
- **想换版本 = 重新发布**：调整 Prompt / 模型 / 权重后，必须再次"发布"才会进入下一局；旧版本对局不受影响。
- **社区引入的快照**：被引入即等价于一次发布，后续若上游有新版本，需用户主动"拉取上游更新"再发布。

> 本地/房间对局不强制走发布流程（用当前编辑态配置即可），发布机制仅用于服务器托管的 Rank 队列。

## 三、对局形态

### 3.A 本地多 Agent 对局（Desktop 专属）

#### 3.A.1 定位
研究者主战场。游戏跑在本机 Bevy，可离线、可不登录，强调**深度调试**与**大日志留存**。

#### 3.A.2 配置与运行
- 双阵营（Order / Chaos）编排，每阵营任意数量槽位（详见 §4 编排原则）
- 启动后进入调试视图，游戏运行在本机；如已登录可手动同步配置到云端
- **大日志留存**：按 GB 量级保存游戏日志到本地（SQLite DB 文件），用于调试游戏与 Agent；日志文件可被日志系统重新加载分析

#### 3.A.3 实时调试面板
- **连接状态**：WebSocket 通道连接状态指示
- **控制台日志**：游戏引擎运行日志，支持按实体 ID 过滤
- **AI 决策思维**：LLM Agent 的完整对话历史流，实时滚动更新
- **RL 指标面板**：强化学习 Agent 的 reward、loss 等实时图表

#### 3.A.4 调试控制（仅本地可用）
- **上帝模式**：无敌 + 全视野
- **无冷却时间**：所有技能冷却归零
- **暂停/恢复**：冻结游戏时间
- **重置位置**：将英雄传送回初始位置
- **切换英雄**：调试视角切换到不同英雄
- **命令行测试床**：手动发送 WebSocket 指令调试

> 这些是"作弊级"调试开关，仅存在于本地对局；房间/Rank 对局一律禁用。

---

### 3.B 房间多用户对局

#### 3.B.1 定位
社交娱乐场景。多名用户聚在一个房间，各自配置 Agent，实时观战 AI 对打。

#### 3.B.2 房间创建与加入（双入口）
- **邀请加入**：房主创建房间 → 分享链接 / 邀请码 → 其他人凭码加入
- **大厅加入**：房间出现在公开大厅列表，任何人可直接加入（房主可关闭大厅可见）
- 两种方式都支持，房主可在创建时选择是否公开到大厅

#### 3.B.3 房间配置（房主控制）
房主在创建房间时设定以下约束：

| 配置项 | 说明 | 默认 |
|---|---|---|
| 最大人数 | 房间成员上限 | 10 |
| 每人 Agent 数上限 | 每个成员最多添加几个 Agent | 3 |
| 阵营策略 | `单阵营`（每人只能在 Order 或 Chaos 一方配 Agent）/ `自由`（不限制） | 自由 |
| 大厅可见 | 是否出现在公开大厅 | 是 |
| 提示词/模型可见 | 成员/观众能否查看他人 Agent 的 Prompt、模型配置 | 否 |

#### 3.B.4 房间内编排
- 每个成员可在房间内**自由添加自己的 Agent，不受阵营限制**（红方配一个、蓝方配两个均可，受房主"阵营策略"约束）
- 编排视图实时展示所有成员添加的 Agent 槽位、所属用户、阵营
- 房主可踢人、可随时调整约束

#### 3.B.5 开局与观战
- **默认进入实时观战**：房主点开始后，全员进入观战态
- **可随时离开**：成员离开不影响对局进行（Agent 决策逻辑跑在服务器上，Agent 自己打）
- **观战形态**：服务器下发各 Agent 操作流 + 日志，客户端同步计算渲染（详见 §5）

#### 3.B.6 异常处理（BYO Agent 掉线）
成员提供的 Agent 若动力源（用户自带的模型 API / RL 推理服务）不响应：
- 对局**立即暂停**
- 全员可见异常指示：**明确显示是哪位用户的哪个 Agent 出现异常**
- 等待该 Agent 恢复，或由房主将其踢出 / 替换

> 平台模型驱动的 Agent 不存在此问题（动力源由平台保障）。

### 3.C Rank 竞技

#### 3.C.1 定位
竞技场景。Agent 作为独立"选手"7×24 全自动排队上分，用户是电竞经理，打磨 Agent 冲榜。

#### 3.C.2 模式（初期：上单 SOLO）
当前仅开放**上单 SOLO** 模式：

| 字段 | 值 |
|---|---|
| 地图范围 | 上路 |
| 人数 | 1v1 |
| 胜利条件 | **先达成任一即胜**：拿一血 / 推掉对方一塔 / 补刀满 100 |
| 出生点 | 由模式规则固定（上路一塔前方） |

> 后续将逐步开放中单 SOLO、下路 2v2、全图 5v5 等模式。每个 Agent 在每个模式下独立 ELO。

#### 3.C.3 全自动匹配
- **7×24 自动排队**：Agent 一旦发布参赛快照并开启排队，即进入匹配池
- **打完立刻排下一局**：每局结束后立即重新匹配，无需用户在线
- 匹配按 ELO 接近原则进行；用户离线时 Agent 仍持续上分

#### 3.C.4 掉线与中止（与房间不同）
Rank 是全自动队列，不能像房间那样无限期暂停等待：
1. 某方 Agent 动力源不响应 → **对局暂停，重试一段时间**（宽限 30 秒）
2. 宽限内恢复 → 继续该局
3. 宽限内仍无法提供操作 → **中止此局，不判胜负**，双方各自回到匹配池进入下一局
4. 频繁中止的 Agent 会被暂时移出队列，待其动力源恢复后再放回

> 平台模型驱动的 Agent 不存在此问题。

#### 3.C.5 排行榜
- **总排行**：历史累计 ELO 全局排名
- **日排行**：当日 ELO 增量排名
- 均按"Agent × 模式"维度展示（如"上单 SOLO 总榜"）

#### 3.C.6 Agent 来源
- 默认用自己发布参赛的 Agent
- 也可从社区引入他人分享的 Agent 快照作为自己的 Agent 参赛（§2.4）

## 四、共享子系统（三形态共用）

### 4.1 设计原则：编排页只做编排，不做内联编辑

对局配置页（本地 / 房间共用）的职责是**从已定义好的资产中"编排"出一局对战**，而非逐项配置细节：

- **编排页保持轻量聚焦**：只展示"谁打谁、从哪出生、怎么算赢"这一层信息，不展开 Prompt、模型权重、超参数等深度字段。
- **复用走"预设 + 下拉选择"**：可复用资产统一封装为预设，在编排页通过下拉框选择。
- **下拉框内置"新建"入口**：每个预设下拉框末尾都有 **＋ 新建预设** / **管理预设**，点击跳转到对应**独立管理页**深度编辑。这是创建/编辑预设的唯一入口。
- **层级关系**：编排页槽位 → Agent → { 英雄 + Agent 配置（脑）+ 出生点预设（坐标）}。Agent 配置与出生点预设不直接在编排页选择，而是由 Agent 组合引用。

> **例外：编排页槽位允许"临时覆盖"**——拖入 Agent 后，仍可在槽位上**临时改写**其绑定的 Agent 配置 / 出生点预设（用于这一局调试，不污染原资产）。改动任一项后槽位即与原 Agent **解绑**（仅保留展开后的具体值），并提供「存为新 Agent」按钮把这份临时配置固化成新 Agent。原资产永不被原地修改。

### 4.2 四类共享资产

| 资产 | 在编排页的角色 | 深度配置位置 |
|---|---|---|
| **Agent** | 每个阵容槽位的**唯一选择单元**（§2.1） | Agent 管理页 |
| **Agent 配置**（"策略大脑"） | 被 Agent 引用为大脑（§2.1） | Agent 配置管理页 |
| **出生点预设** | 被 Agent 引用为坐标（Rank 模式由规则覆盖） | 出生点预设管理页 |
| **胜利条件** | 选择预设模板或自定义 | 胜利条件构建器 |

### 4.3 Agent 与 Agent 配置

- **Agent 配置**（英雄无关的大脑）：类型、Prompt、模型/权重、前言等；以 JSON 持久化，支持导入/导出分享
- **Agent**：命名、英雄、引用的 Agent 配置、引用的出生点预设；以 JSON 持久化，支持导入/导出
- **临时覆盖**：编排页槽位拖入 Agent 后，Agent 配置 / 出生点选择器始终展开，初始显示继承值并打「继承」标记；改动任一项 → 槽位与原 Agent **解绑**，标记变「已覆盖」，原 Agent 不被修改；解绑后出现「存为新 Agent」按钮（自动命名 `<原名> · 副本`，重名追加 `· 副本 2 / 3 / …`）

### 4.4 出生点预设

- 统一封装为预设，编排页通过下拉框选择，不内联输入坐标
- 内置常用位置：上路一塔前方、中路河道、下路草丛、蓝方 Buff 区等
- **出生点预设管理页**负责全部 CRUD：通过 SVG 地图点选器交互式选取坐标（15000×15000 游戏坐标系）、命名保存、按阵营（Order / Chaos）筛选

### 4.5 胜利条件配置

支持灵活的胜利条件系统，可自由组合形成复杂判定逻辑。

#### 基础条件原子

| 条件类型 | 参数 | 示例 |
|---|---|---|
| 补刀数达标 | `minion_kills >= N` | 谁先补 10 个小兵 |
| 击杀英雄 | `kills >= N` | 拿下一血 / 三杀获胜 |
| 摧毁防御塔 | `turret_destroyed(lane, tier)` | 推掉上路一塔 |
| 摧毁水晶 | `inhibitor_destroyed(lane)` | 摧毁任意水晶 |
| 摧毁基地 | `nexus_destroyed` | 推掉主基地 |
| 经济领先 | `gold_lead >= N` | 经济差达到 2000 |
| 存活时长 | `survive_duration >= T` | 存活超过 5 分钟 |
| 总伤害量 | `total_damage >= N` | 累计输出 10000 伤害 |

#### 逻辑组合
- **AND**：所有子条件同时满足
- **OR**：任一子条件满足
- **NOT**：条件取反
- 支持嵌套组合，形成任意深度的胜利条件树

#### 预设胜利条件
- **速战速决**：先拿一血获胜
- **补刀竞赛**：先补 10 个小兵获胜
- **推塔竞速**：先推掉对方一塔获胜
- **综合实力**：推掉主基地获胜（标准模式）
- 用户可创建自定义胜利条件预设

### 4.6 对局场景（Scenario）

- 完整对局配置（阵营 Agent + 出生点 + 胜利条件）可保存为**场景预设**
- 场景以命名持久化，支持从侧边栏快速加载
- 支持从历史对局一键复刻场景配置

## 五、观战与回放

### 5.1 观战形态：操作流 + 日志，客户端渲染

所有服务器托管对局（房间 / Rank）采用**操作流同步**而非视频流：

- **服务器下发**：各 Agent 的操作流 + 结构化日志
- **客户端同步计算渲染**：客户端拿到操作流后本地重放并渲染（Bevy 渲染，未来 Web 端走 WASM/WebGPU）
- **优点**：带宽低、可任意视角观战、可暂停回拖

### 5.2 日志留存

- **服务器仅保留 24 小时**的日志（磁盘成本约束）
- **日志可下载**：用户可下载完整日志（SQLite DB 文件）到本地
- **日志 DB 可加载**：日志系统支持加载本地/下载的 DB 文件，在 Desktop 调试面板中分析回放

### 5.3 回放（事件流）
- 回放基于**各 Agent 的操作流**，确定性重放整局
- 房间/Rank 历史对局可随时回看；完整对话历史（LLM Agent 的思考链）持久化存储

### 5.4 隐私
- 观战时他人 Agent 的 Prompt、模型配置等敏感字段**默认不可见**
- 房间由房主设置可见性；Rank 公开对局的敏感字段一律不可见

## 六、算力、精粹与计费

### 6.1 服务器算力约束

- 单局对局约 **50 MB 内存**占用（实测）
- 服务器并发对局数受物理内存上限约束；需要开发**观测与管理后台**用于监控各对局资源占用、调度、熔断

### 6.2 模型动力源（两条路径）

| 路径 | 说明 | 计费 |
|---|---|---|
| **平台模型** | 平台内置多种模型（Gemini / DeepSeek 等），用户直接选用 | 按实际 Token 用量，以**精粹**结算 |
| **BYO 模型** | 用户自行提供模型 API（填自己的 Key / Base URL）作为 Agent 动力 | 不走平台模型，不消耗精粹 |

> RL 推理服务同样属于 BYO：用户自行部署推理端点，平台只提供接入接口。

### 6.3 精粹（Blue Essence）

**来源**（仅两种）：
- 每日签到
- 充值

**用途**：
- 抵扣平台模型的 Token 消耗
- 购买额外 Agent 槽位（见 §6.4）

### 6.4 订阅与 Agent 槽位

- **免费用户**：最多创建 **5 个 Agent**，模式不限，对局局数不限
- **扩槽**：花精粹购买额外 Agent 槽位
- **阶梯制订阅**：订阅后提供一定量精粹 + 更多 Agent 槽位等增值服务（档位待定）

### 6.5 防滥用
- Rank 全自动排队受平台算力调度约束，可能排队等待
- BYO 动力源的 Agent 频繁掉线会被暂时移出队列（§3.C.4）

### 8.2 关键技术栈

| 层 | 技术 |
|---|---|
| 游戏引擎 | Bevy 0.17（Rust ECS），自定义分支支持 WebGPU + Basis Universal |
| Web 服务端 | Axum + SQLx + PostgreSQL + JWT（手机号鉴权） |
| 桌面壳 | Tauri 2.x（Rust backend + WebView frontend） |
| 前端 | Vue 3 + TypeScript + Pinia + TailwindCSS + shadcn-vue |
| 通信 | WebSocket（Bevy ↔ Web Server ↔ 前端；WS 反向代理） |
| LLM 集成 | Anthropic API（通过 Rig SDK），支持 Tool Use |
| RL 训练 | Python + PyTorch + CleanRL (PPO) + Gymnasium |
| 资源解析 | 自研 WAD/PROP/TEX/SKN/SKL/ANM 解析器 |

### 8.3 后端服务现状（lol_web_server）

`crates/lol_web_server` 已完成服务化骨架，接口分层为 trait（`ConfigService` / `PresetService` / `ScenarioService` / `GameService` / `HistoryService` / `LogService` / `UserService`）+ 实现，支持用户数据隔离（所有数据按 `user_id` 分域）：

- **Auth**：`/api/auth/{register,login,reset-password}`（JWT）
- **AI Config**：`/api/config`（BYO 模型 Key/URL）
- **Presets**：spawn / agent / hero 三类预设 CRUD
- **Scenarios**：自定义场景 + 胜利条件存取
- **Game**：`/api/game/{start,stop}`
- **Histories**：对局历史列表/详情/删除
- **Logs**：实体/分类/查询/清空
- **WS 反向代理**：`/api/ws` → `ws://127.0.0.1:<game_port>`，前端 WS 经 web server 转发到对局 Bevy

> 待扩展：房间/大厅、Rank 匹配队列、参赛快照、社区 Agent、精粹/订阅、算力监控后台。
