# 对局与房间/Rank/观战系统 — 产品设计

## 一、对局形态总纲

平台支持三种对局形态，**长期并存、各有侧重**，且共享同一套 Agent 资产（预设库、配置、社区）：

| 形态 | 游戏运行位置 | 登录 | Agent 来源 | 侧重点 |
|---|---|---|---|---|
| **A. 本地多 Agent 对局** | **本机**（desktop 独占） | 可不登录 | 本地预设库 | 研究调试 |
| **B. 房间多用户对局** | 服务器 | 必须登录 | 各成员自带 Agent | 社交娱乐 |
| **C. Rank 竞技** | 服务器 | 必须登录 | 各自发布的参赛快照 | 竞技排名 |

### 1. 形态演进与依赖关系

```
本地调试（desktop 独占，可离线）
        │
        │  共享 Agent 资产 / 服务器化
        ▼
房间多用户对局（社交，实时观战）
        │
        │  复用房间的对局编排 / 服务器托管能力
        ▼
Rank 竞技（全自动 7×24 排队上分）
```

- **房间是 rank 的基础**：rank 本质上是"系统当房主、全自动匹配的房间"，复用同一套服务器对局托管与观战管线。
- **Desktop 是 Web 的超集**：Web 端是轻量消费端；Desktop 保留为"专业调试版"，Web 能做的（房间/rank/观战）Desktop 都能做，额外拥有本地离线调试能力。

## 二、三种对局形态详情

### 3.A 本地多 Agent 对局（Desktop 专属）
- **定位**：研究者主战场。游戏跑在本机 Bevy，可离线、可不登录，强调**深度调试**与**大日志留存**。
- **配置与运行**：双阵营（Order / Chaos）编排，每阵营任意数量槽位。启动后进入调试视图。
- **大日志留存**：按 GB 量级保存游戏日志到本地（SQLite DB 文件），用于调试游戏与 Agent；日志文件可被日志系统重新加载分析。
- **调试控制（仅本地可用）**：
  - **上帝模式**：无敌 + 全视野
  - **无冷却时间**：所有技能冷却归零
  - **暂停/恢复**：冻结游戏时间
  - **重置位置**：将英雄传送回初始位置
  - **切换英雄**：调试视角切换到不同英雄
  - **命令行测试床**：手动发送 WebSocket 指令调试

### 3.B 房间多用户对局
- **定位**：社交娱乐场景。多名用户聚在一个房间，各自配置 Agent，实时观战 AI 对打。
- **房间创建与加入（双入口）**：
  - **邀请加入**：房主创建房间 → 分享链接 / 邀请码 → 其他人凭码加入
  - **大厅加入**：房间出现在公开大厅列表，任何人可直接加入
- **房间配置（房主控制）**：房主设定最大人数（默认 10）、每人 Agent 数上限（默认 3）、阵营策略（单阵营/自由）、大厅可见性、提示词/模型可见性等约束。
- **房间内编排**：每个成员可在房间内自由添加自己的 Agent，不受阵营限制。
- **开局与观战**：房主点开始后，默认全员进入实时观战。成员中途离开不影响对局在服务器上继续进行。
- **异常处理（BYO Agent 掉线）**：成员提供的 Agent 若动力源不响应，对局立即暂停，且全员可见异常指示。

### 3.C Rank 竞技
- **定位**：竞技场景。Agent 作为独立选手 7×24 全自动排队上分，用户是电竞经理。
- **模式（初期：上单 SOLO）**：
  - 地图范围：上路 1v1。
  - 胜利条件：先达成任一即胜：拿一血 / 推掉对方一塔 / 补刀满 100；若游戏超过 15 分钟仍未分胜负，则按补刀数判定胜负（多者胜）。
  - 出生点：由模式规则固定（上路一塔前方）。
- **全自动匹配**：7×24 自动排队，打完立刻重新排队，匹配按 ELO 接近原则进行。
- **掉线与中止（与房间不同）**：某方 Agent 动力源不响应，对局暂停并重试（宽限 30 秒）。宽限内未恢复则中止此局，不判胜负，各自重新匹配。频繁中止的 Agent 会被暂时移出队列。
- **排行榜**：分总排行榜（历史累计 ELO）与日排行榜（当日 ELO 增量），按“Agent × 模式”维度展示。

## 三、共享子系统（三形态共用）

### 1. 设计原则：编排页只做编排，不做内联编辑
对局配置页的职责是**从已定义好的资产中"编排"出一局对战**，保持轻量聚焦。复用走"预设 + 下拉选择"（包括选手下拉和出生点下拉，两者解耦，互不干扰），下拉框内置"管理"入口供跳转深度编辑。

### 2. 出生点预设
- 统一封装为预设，编排页通过下拉框选择。
- 出生点预设管理页负责全部 CRUD：通过 SVG 地图点选器交互式选取坐标（15000×15000 游戏坐标系）、命名保存。

### 3. 胜利条件配置
支持灵活的胜利条件系统，可自由组合形成复杂判定逻辑：
- **基础条件原子**：补刀数达标 (`minion_kills >= N`)、击杀英雄 (`kills >= N`)、摧毁防御塔 (`turret_destroyed`)、摧毁水晶/基地、经济领先、存活时长、总伤害量等。
- **逻辑组合**：支持 AND、OR、NOT 嵌套组合，形成任意深度的胜利条件树。
- **预设模板**：速战速决（一血）、补刀竞赛（10小兵）、推塔竞速（一塔）、综合实力（基地）。

### 4. 对局场景 (Scenario)
完整对局配置（阵营 Agent + 出生点 + 胜利条件）可保存为命名**场景预设**，支持从侧边栏快速加载或从历史对局一键复刻。

## 四、观战与回放

### 1. 观战形态：操作流 + 日志，客户端渲染
所有服务器托管对局采用**操作流同步**而非视频流：
- **服务器下发**：各 Agent 的操作流 + 结构化日志。
- **客户端同步计算渲染**：客户端拿到操作流后在本地重放并渲染。Web 端通过 `lol_render` 的 WebGPU/WASM 画布挂载渲染；Desktop 端直接起游戏进程渲染，不走画布。
- **优点**：带宽低、视角自由、可暂停回拖。

### 2. 日志留存
- 服务器仅保留 24 小时的日志。
- 支持将一局的完整日志打包为 SQLite DB 文件并提供下载，可在本地 Desktop 调试面板中加载分析与回放。

### 3. 隐私
- 观战时他人 Agent 的 Prompt、模型配置等敏感字段默认不可见。
