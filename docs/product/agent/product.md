# Agent 选手与决策体系 — 产品设计

## 一、核心术语定义与区分：英雄、选手 (Agent)

为避免概念混淆，产品文档统一并明确区分以下概念：

1. **英雄 (Hero)**
   - **定义**：游戏中的具体角色（如：`Riven`、`Fiora`），决定了实体的技能机制（如 Q/W/E/R）、基础属性（如生命值、攻击力、防御力）和游戏内的 3D 物理模型。
   - **性质**：是游戏引擎内的核心战斗单元，只提供物理和技能的承载，不具备自主决策能力。

2. **选手 (Agent / 预设)**
   - **定义**：将一个具体的 **英雄 (Hero)** 与特定的 **决策类型**、**策略配置**（LLM 提示词、模型、前言，或 RL/Script 的 JSON 配置）融为一体的完整“选手”预设。
   - **性质**：是参加对局、拥有独立 ELO 评分、出现在排行榜上的主体。用户在选手管理页（“我的选手”）直接配置其全部运行策略，而无需在多个层级间跳转配置。在自定义对局（本地/房间对局）中，出生点由槽位独立指定，不绑定于选手本身。
   - **例**：「锐雯 · 激进压制」（LLM 决策，携带激进 Prompt）、「菲奥娜 · 稳健补刀」（LLM 决策，携带稳健 Prompt）。

---

### 核心术语关系图

```
┌─────────────────────────────────────────────────────────────┐
│  选手 (Agent) = 英雄 + 决策类型 + 策略配置 (LLM/RL/Script)    │
│    例：「锐雯 · 激进压制」「菲奥娜 · 稳健补刀」              │
└─────────────────────────────────────────────────────────────┘
                           │ 发布（冻结）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  参赛快照 (Snapshot) = Agent 在某时刻的不可变冻结版本        │
│    用于：Rank 队列参赛 / 分享到社区被他人引入                │
└─────────────────────────────────────────────────────────────┘
```

**关键规则**：
- **ELO 主体是 Agent**，不是用户。用户是电竞经理，可养多个 Agent。
- ELO 按 **Agent × 模式** 分别计算（同一 Agent 的上单 ELO 与中单 ELO 互不影响）。
- **进行中的对局使用参赛快照**：用户修改选手配置不影响进行中的局，下次发布/参赛才生效。
- **一个 Agent 可报名多个模式**（上单 SOLO、未来中单 SOLO……），各模式独立 ELO。

## 二、Agent 类型

支持三种 Agent 类型，覆盖从规则驱动到智能决策的完整光谱。策略配置按类型携带不同字段：

| 类型 | 驱动方式 | 适用场景 |
|---|---|---|
| **LLM Agent** | 大语言模型（Gemini / DeepSeek 等） | 自然语言策略推理、复杂决策、探索性实验 |
| **RL Agent** | 强化学习模型（PPO 等） | 高频微操、补刀优化、机械性任务 |
| **Script Agent** | 用户编写的 JavaScript 脚本 | 确定性行为树、固定套路、基准测试对手 |

## 三、各类型 Agent 的运行时能力

### LLM Agent
- 通过 WebSocket 接收游戏观测数据（`get_observe`），调用 LLM 推理后下发动作指令（`action`）
- Desktop 端实时展示 Agent 的**思考链（Chain of Thought）**，以对话气泡的形式呈现：
  - 🟢 **系统观测**（User 消息）：当前局势数据摘要
  - 🔵 **模型推理**（Assistant 消息）：思考过程、决策依据、最终动作
- 支持配置系统级 Preamble Prompt（全局策略指导）和英雄级 Prompt（角色个性化微调）
- 模型选择为「供应商 + 模型名」级联下拉，供应商与模型列表来自设置页的模型供应商配置（含智谱、DeepSeek 等预设及自定义供应商），不再手填；可调节思考深度

### RL Agent
- 通过 Gymnasium 环境接口（`MoonLoLEnv`）与游戏引擎交互
- Desktop 端展示训练/推理过程中的关键指标：
  - **Reward 分项**：补刀奖励、击杀奖励、死亡惩罚、位置奖励等各分项 reward
  - **策略分布**：当前动作空间的概率分布可视化
  - **Value 估计**：Critic 网络对当前状态的价值评估
  - **训练曲线**：Episodic Return、Loss、KL 散度等实时图表
- 支持加载已训练的模型权重（`.pth`）；**在线训练由用户自行进行**，平台只提供接入接口作为 Agent 动力

### Script Agent
- 使用 **JavaScript** 作为脚本语言，通过内嵌 JS 运行时（如 Boa / QuickJS）在 Rust 侧执行
- 内置脚本编辑器（Monaco Editor），原生支持 JS 语法高亮、智能补全和类型提示
- 支持脚本的**实时热重载**：修改脚本后无需重启对局
- 提供调试工具：断点、单步执行、变量观察
- 暴露游戏 API 绑定（`observe()`、`action()`、`log()` 等），脚本可直接调用
- 内置常用脚本模板：补刀脚本、走A脚本、技能连招脚本

## 四、Agent 的所有权、可见性与社区

- **全局预设库**：所有选手 (Agent) / 出生点 / 场景统一管理，跨形态复用。选手以**云端为主存储**：桌面端在线时读写云端（写入时镜像到本地缓存，离线降级本地），房间/Rank 对局直接取云端；Web 端始终云端。
- **可见性**（用户设置）：`私有` / `好友可见` / `公开`。
- **Fork 与上游同步**：公开的 Agent 可被他人 Fork 为自己的副本；Fork 后可"从上游拉取最新选手配置"同步更新。
- **社区快照引入**：用户可把某个 Agent 的参赛快照分享到社区，其他用户可引入作为自己的 Agent 参赛。
- **可见性即权限**：导出/查看 Prompt / 模型配置等敏感字段的权限，由该 Agent 的可见性设置决定。

## 五、选手快照与版本（Rank 专用）

Rank 队列里用户可能不在线，Agent 的配置变更必须显式生效，避免"对局进行中配置被改"的不一致：

- **发布（Publish）**：用户手动把 Agent 的当前配置**发布为一个参赛快照**，记为 `v1 / v2 / v3 …`。
- **队列取用规则**：Rank 匹配队列**始终使用该 Agent 最新发布的那一份快照**。
- **想换版本 = 重新发布**：调整 Prompt / 模型 / 权重后，必须再次"发布"才会进入下一局；旧版本对局不受影响。
- **社区引入的快照**：被引入即等价于一次发布，后续若上游有新版本，需用户主动"拉取上游更新"再发布。

> 本地/房间对局不强制走发布流程（用当前编辑态配置即可），发布机制仅用于服务器托管的 Rank 队列。
