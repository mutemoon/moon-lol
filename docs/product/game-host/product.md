# 游戏进程托管 — 产品设计

本文档从产品视角描述 Bevy 游戏子进程托管在桌面端与云端的能力范围、差异与约束。

## 一、定位

游戏进程托管是「起一个 Bevy 游戏实例、管理其生命周期、与之交互」的基础能力。它本身不含对局语义——不记录对局、不判胜负。对局体系是云端竞技场景叠加在它之上的能力。

桌面端只有本地调试模式跑在本地；自定义房间与 Rank 匹配都在云端。因此桌面本地不需要匹配系统 / 胜负判定 / 对局记录，只需要管理多个游戏进程与交互能力。

## 二、两端能力对照

| 能力 | 桌面本地调试 | 云端竞技 |
|---|---|---|
| 多游戏并发 | 端口池分配，多局并存 | 端口池分配，多局并存 |
| 进程启动 | dev cargo run / release 打包二进制，非 headless | cargo run 或 MOON_LOL_BINARY 二进制，headless |
| 进程停止 | 按 id 停止 | 按 match_id 停止 |
| 与游戏交互 | 直连 Bevy WebSocket 实时调试 | 服务端持 WS，前端轮询对局事件 |
| AI 决策环 | start 时自动起，读本地 providers.json | start 时显式 spawn，按 owner_id 解析云端供应商 |
| 对局记录 | 无 | MatchRepo 落库 |
| 胜负判定 | 无 | match_supervisor 套用 SOLO 规则 |
| 窗口渲染 | 回放 / 观战需窗口（非 headless） | 永远 headless |

## 三、桌面本地调试

桌面本地调试面向研究者在单机调试 Agent 策略：起一局带场景 agent 的本地对局，AI 自动决策，研究者通过实时调试通道观测与干预。

- 起多游戏：端口池分配，不再写死端口，可同机多局。
- 实时调试：保留按端口直连 Bevy WebSocket 的通道，下发指令、收取观测，低延迟。
- AI 决策环：start 时自动启动，读本地 `~/.moon-lol/providers.json` 解析凭证。
- 回放 / 观战：非 headless 启动以渲染窗口，不启动 AI 环。

桌面本地不引入对局记录与胜负判定——这些是云端竞技体系，本地调试不需要。

## 四、云端竞技

云端竞技面向房间社交与 Rank 排名，必须登录。对局在服务器运行，前端轻量消费。

- 进程托管复用同一份 `GameProcessManager`。
- 叠加对局体系：match 记录、状态机、match_supervisor 胜负判定、对局事件流落库。
- AI 决策环按每请求的 owner_id 解析云端供应商凭证；选手选平台模型时走平台网关 env，按 Token 消耗以精粹结算。
- 始终 headless，无窗口。

## 五、约束

- 单局 Bevy 实例约 50 MB 内存，并发数受物理内存约束。
- 端口池 9100 至 9200，上限 100 并发游戏进程；池满拒绝新启动。
- 进程状态不持久化：服务 / 桌面重启后，运行中的游戏视为已退出，端口释放。
- spawn 命令构建、端口池、进程表逻辑只此一份（共享 crate），各端只决定 program、前缀、cwd、headless 等配置差异。
