# 技能系统架构设计理念

技能系统已演进为全 **代码驱动 (Code-Driven)** 的原生 **ECS + 观察者 (Observer)** 模式。我们彻底抛弃了静态的数据效果序列，转而使用 Rust 逻辑直接控制技能流，这为复杂控制流（如 Riven 的分段 Q、盲僧的二段 Q）提供了无限的灵活性。

---

## 核心设计原则

### 1. 代码驱动逻辑 (Code-Driven over Data-Driven)

**拒绝“黑盒序列”**：不再通过 `Vec<SkillAction>` 等静态数据来定义技能。每个技能现在由一个专用的 `Observer` 处理 `CommandSkillExecute` 事件。

- **复杂分支**：技能函数内部可以使用完整的 Rust 语法（if/else, match, loop）来处理分段、条件分支和复杂的时序依赖。
- **原子解耦**：利用 `commands.trigger(ActionDash { .. })` 等原子命令快速拼装表现逻辑。

### 2. 利用 ECS 实体承载状态 (Stateful Skill Entities)

每个技能不再只是一个数组索引，而是一个**具有独立状态的实体**。

- **持久化状态**：通过在技能实体上添加 Marker 组件（如 `SkillRivenQ`）或状态组件（如记录 Q 分段的 `RivenQStage`），实现技能逻辑的演进。
- **生命周期透明**：技能的冷却、持续时间、甚至中间状态都可以通过查询实体组件直接观测和调试。

### 3. 解除抽象负担 (Zero-Abstraction Overhead)

移除了行为树（bevy_behave）和中间资产层（SkillEffect），让设计与实现合一。

- **易于 AI 开发**：AI 模型处理标准的 Rust ECS 逻辑远比处理自定义的 DSL 或复杂的 JSON 配置更准确。
- **白盒级调试**：通过 Bevy Inspector 或日志，每一行逻辑的触发都清晰可见，彻底消除“数据配置对了但就是没动静”的玄学 Bug。

### 4. 表现与判定分离 (Action Triggering)

通过 `commands.trigger` 触发标准化的原子动作：

- **Animation**: 触发特定 Hash 的动画。
- **Dash**: 执行位移（支持位移结束触发特定观察者）。
- **Damage**: 在指定时间/位置触发计算精度的伤害判定。
- **Buff**: 动态附加状态，支持高度自定义的 Bundle 注入。

---

## 典型实现范式：Riven Q (折翼之舞)

1.  **分段判定**：`on_skill_riven_q` 观察者被触发。
2.  **状态查询**：查询技能实体上是否存在分段 Buff 或 Marker 组件。
3.  **分支执行**：
    - **Q1/Q2**：触发向前位移、播放 A/B 动画、造成圆形伤害、并在技能实体上插入下一段的 Marker。
    - **Q3**：触发带击飞的冲击波、播放 C 动画、清理分段状态。
4.  **冷却管理**：逻辑执行完毕后，主系统统一处理冷却时钟。

## 优势总结

| 维度         | 旧架构 (Data-Driven / Behave)      | 新架构 (Code-Driven ECS)             |
| :----------- | :--------------------------------- | :----------------------------------- |
| **灵活性**   | 难以处理复杂的条件流和时序逻辑     | 拥有 Rust 全部的表达能力             |
| **可观测性** | 行为树执行流难以透视               | ECS 组件状态一目了然                 |
| **AI 协作**  | 经常产生错误的配置数据或树节点定义 | 极高概率生成正确的系统或观察者代码   |
| **复用性**   | 依赖特定的效果节点                 | 基于原子组件和命令，真正的积木化复用 |
| **稳定性**   | 容易出现状态残留和打断异常         | 随实体销毁自动彻底清理               |
